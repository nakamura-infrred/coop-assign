# 開発の哲学と設計思想

このドキュメントは、本プロジェクトの根底にある考え方をまとめたものです。機能追加やリファクタリングの際、迷ったときの指針としてください。

## 1. 目的： 割当担当者の「右腕」を作る

私たちの最優先顧客は、膨大な調整作業を一身に背負っている**「割り振り担当者」**です。

- **目指すこと**: 担当者が「誰が空いているか」「どこが重複しているか」を調べる時間を半分以下にすること。
- **目指さないこと**: コミュニティ全体の合意形成プラットフォームを作ることや、全自動で割当を完結させること（人間の判断を奪わない）。

## 2. デザインの原則： 「レジリエンス（業務継続性）」

システムはいつか止まる、あるいは使われなくなる可能性があります。その時でも、現場の業務が止まらないことが重要です。

- **データは外に出せる**: 常に Google Sheets や CSV に最新情報を書き出せる、あるいは印刷して持ち出せることを機能の「おまけ」ではなく「核心」と位置づけます。
- **システムへの依存を最小限に**: 本システムがなくても「やり方」がわかるような、現場のアナログ運用に寄り添ったデータ構造と UI を採用します。

## 3. 実装の指針： 「疎結合とシンプルさ」

- **過度な自動化を避ける**: 複雑なアルゴリズムよりも、担当者がマニュアルで操作する際のストレスを減らす改善（ドラッグ＆ドロップ、即時バリデーション）を優先します。
- **メンテナンス性**: OSS として、あるいは個人のボランティアが引き継いでも理解できる、シンプルで標準的な技術選定（React, Firebase）とコードを維持します。

## 4. なぜ「共助」という言葉があったのか

初期の表現にあった「共助」とは、コミュニティ全員がシステムに精通することではなく、**「担当者が一人で抱え込んでいた情報を可視化することで、結果的に周囲が協力しやすくなる（あるいは担当者の負担が周囲に見えるようになる）」**という状態を指しています。

主体はあくまで担当者であり、ツールはその負担を肩代わりするためのものです。
